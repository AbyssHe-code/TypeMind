# 功能列表同步 Workflow

> **用途**：每次前端应用经过一轮迭代后，运行此 workflow 将功能列表文档与当前实现对齐。 
> **触发时机**：用户主动出发

## 输入清单

运行前确认以下文件已就位：

| # | 文件 | 说明 | 来源 |
|---|------|------|------|
| 1 | `Function-List_vX.md` | 最新一版功能列表（X = 当前最大版本号） | 需求资料/contexts |
| 2 | 前端应用代码 | 项目目录下的前端应用代码 | 根目录文件夹 |

若用户未上传前端代码，在开始 Step 2 前停下来请求：
> 「请把当前的前端文件上传给我（单文件 HTML 或项目压缩包均可），我来分析实现了哪些功能。」

---

## 执行步骤

### Step 1 · 读取现有功能列表

```
Read: 需求资料/contexts/Function-List_vX.md
```

提取并在内存中保留以下结构：

```
{
  version: "vX",
  modules: [
    {
      id: "模块编号",
      name: "模块名称",
      priority: "P0/P1/P2",
      features: [
        { id: "功能编号", name: "功能名称", description: "..." }
      ]
    }
  ],
  roadmap: { mvp: [...], phase2: [...], phase3: [...] },
  techStack: { ... }
}
```

---

### Step 2 · 分析前端代码

读取前端项目文件，按以下维度逐一核查。**只记录代码中实际存在的功能，不做假设。**

#### 2-A 页面 / 路由清单

扫描：
- `<Route path=...>`（React Router）
- `view === '...'` 的状态切换（单文件 SPA）
- `navigate(...)` / `router.push(...)` 的目标路径

输出一张表：

| 路由 / 视图 | 对应页面 | 实现状态 |
|------------|---------|---------|
| `/` 或 `view='home'` | 首页 | ✅ 已实现 |
| ... | ... | ... |

#### 2-B 功能实现清单

对每个页面，细读代码并回答：

**认证 & 账户**
- [ ] 注册页（用户名 + 密码）是否存在？
- [ ] 登录页是否存在？
- [ ] 表单校验逻辑（用户名格式、密码强度）？
- [ ] 已登录态的保持方式（localStorage / sessionStorage / cookie）？

**测评模块**
- [ ] 题目数量？（设计稿：20 题，实际几题？）
- [ ] 维度覆盖？（E/I · S/N · T/F · J/P 各几题？）
- [ ] 结果页呈现什么？（类型名 / 认知功能栈 / 雷达图 / 维度得分条）
- [ ] 是否支持"已知类型快速入驻"？

**AI 对话模块**
- [ ] 场景入口数量？（设计：6 个）
- [ ] 对话输出结构？（是否包含三段式：根因 / 对方视角 / 可操作建议）
- [ ] 情绪接收首句机制是否存在？
- [ ] 是否有对方类型推断功能（AI 追问行为特征）？
- [ ] 记忆系统：读写了哪些数据？存在哪里？

**内容模块**
- [ ] 场景解读库是否实现？几个场景？
- [ ] 类型冲突图谱是否实现？

**档案 & 关系**
- [ ] 个人档案页是否存在？
- [ ] 同事 / 关系档案功能是否存在？
- [ ] AI 对我的理解摘要是否存在？

**历史 & 分享**
- [ ] 对话历史页是否存在？
- [ ] 分享功能（图片卡片）是否存在？

#### 2-C 与功能列表的差异对比

逐模块对比"设计规划"与"实际代码"，分三类标记：

| 类别 | 说明 | 符号 |
|------|------|------|
| 已实现 | 代码中存在且可用 | ✅ |
| 已实现但与规划有偏差 | 存在但细节不同（如题目数量、字段名） | ⚠️ |
| 未实现 | 规划中有，代码中无 | ❌ |
| 新增（规划外） | 代码中有，功能列表里没有 | 🆕 |

---

### Step 3 · 生成新版功能列表

基于 Step 2 的分析结果，生成 `Function-List_v(X+1).md`。

**生成规则：**

1. **保留不变的内容**：产品定位、北极星指标、核心价值主张、与竞品的本质差异对比表 — 这些属于产品战略层，不随代码变化而改变，除非用户明确要求修改。

2. **更新功能描述**：
   - 已实现（✅）：将状态改为"已实现"，描述与代码实际行为对齐
   - 有偏差（⚠️）：在功能描述后注明"当前实现与规划略有差异"，并说明实际行为
   - 未实现（❌）：保留规划描述，标注"待开发"
   - 新增（🆕）：在相应模块下增加新功能条目，标注"新增 · 未在原规划中"

3. **优先级重评估**：
   - 已实现的功能，优先级自动调整为"✅ 已上线"（不再用 P0/P1/P2）
   - 未实现的功能维持原有优先级
   - 新增功能默认标注"P? 待评估"

4. **路线图更新**：
   - 将已完成功能从对应阶段中标记为"✅ 完成"
   - 将实际新增功能加入路线图

5. **版本信息更新**：
   - 文档顶部版本号递增（vX → vX+1）
   - 更新日期为今天
   - 新增一行"上一版本"指针：`> 基于 vX 更新，对应代码快照日期：YYYY-MM-DD`

**文件命名**：`Function-List_v(X+1).md`

---

### Step 4 · 输出变更摘要

在对话中（不写入文件）输出一份简短的变更摘要，格式如下：

```
## 本次更新摘要（v{X} → v{X+1}）

### 新实现的功能（✅）
- [模块名] 功能名 — 简述实际实现方式

### 与规划有偏差的功能（⚠️）
- [模块名] 功能名 — 偏差说明

### 规划中尚未实现（❌）
- [模块名] 功能名

### 规划外新增功能（🆕）
- [模块名] 功能名 — 简述

### 建议
（如果发现产品方向偏移或重大实现问题，在此简短说明）
```

## 常见场景处理

### 场景 A：用户上传了单文件 HTML
直接 `Read` 该文件，从头扫描到尾，重点关注：
- 所有 `view === '...'` 的状态分支
- 所有函数定义（`function renderXxx()` 或组件名）
- `localStorage` 的读写 key
- fetch / axios 调用的接口 URL

### 场景 B：用户上传了 JSX/React 项目（压缩包）
1. 用 bash 解压：`unzip uploads/xxx.zip -d /tmp/project`
2. 先看 `src/App.jsx` 或 `src/router.jsx`，获取路由结构
3. 再按路由逐一 `Read` 对应页面组件
4. 看 `src/store/` 或 `src/context/` 了解状态管理与记忆系统

### 场景 C：用户只描述了改动，没有上传代码
不直接生成新功能列表。回复：
> 「为了准确反映当前实现，我需要读取最新的前端代码文件。请上传当前版本的代码（HTML 单文件或项目压缩包）。」

### 场景 D：功能列表版本号不确定
扫描 `需求资料/contexts/` 目录，找版本号最大的 `Function-List_vX.md` 作为基础文档。

## 质量检查清单

生成新版功能列表前，逐项确认：

- [ ] 产品北极星和定位未被无意改动
- [ ] 每个模块的优先级状态（✅/⚠️/❌/🆕）标注完整
- [ ] 新增功能有清晰的来源说明（"来自当前实现"）
- [ ] 路线图中已完成项已标记
- [ ] 版本号和日期已更新
- [ ] 文件已保存至 需求资料/contexts/ 并通过 present_files 展示给用户

---

## 示例调用

用户说：
> 「更新功能列表。」（同时上传了新的前端文件）

执行：
1. `Read 需求资料/contexts/Function-List_v1.md` → 提取模块结构
2. `Read src/[前端文件]` → 逐维度分析
3. 生成差异对比表
4. `Write 需求资料/contexts/Function-List_v2.md`
5. `present_files` 展示文件
6. 在对话中输出变更摘要
